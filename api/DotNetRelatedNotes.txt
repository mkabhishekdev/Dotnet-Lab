6. "async"/"await"
       - As known, this is used for asynchronous messaging
       - In simpler words, its like making the data retrieval from database 'faster'. Usually in a call
         to any database server or even network calls(like in microservices) when the call goes from the 
         code to these outside servers. Usually since they are external to the code the calls usually will
         take time. "Async" concept got first introduced into .net and later it spread to other platforms.
       - Adding "async" to method declaration is the starting point, the actual retrieval logic happens
         only when the "await" keyword is used in the calls to the database/network servers within the 
         methods.
       - When .NET encounters a "await" keyword, it will do something called a 'co-routine' i.e co-routine
         means you will get back the result in the form of a 'yield' & this will allow the code to interrupt
         any other processes which are going on. Than it will quickly return -> after that you can go to 
         your next piece of code.
       - So add 'await' whenever you are hitting the database (In most situations)

8. UNDERSTANDING: Singleton vs Scoped vs Transient
   ************************************************

  RESTAURANT PLATE EXAMPLE(Analogy):

  🍽 Scoped (plate of food per request)

    - Each customer (HTTP request) gets one plate.
    - During your meal (whole request), the same plate is used.
    - When a new customer arrives (new request), a new plate is given.
  👉 This is how DbContext usually works in ASP.NET Core.

  🍽 Singleton (one plate for the whole restaurant)

    - Imagine the restaurant has only one plate in the entire building.
    - Every customer, no matter when they come, eats from that same single plate.
    - The plate never changes until the restaurant shuts down.
  👉 That’s how a Singleton service works — one instance for the whole app lifetime.

  🍽 Transient (new plate every time you ask)

    - Every time you ask the waiter for something, you get a brand new plate, even in the middle of your meal.
    - If you ask 5 times during one meal → you’ll get 5 plates.
    - Each request, each usage → always a new plate.
  👉 That’s how a Transient service works — always a fresh instance.

  CODE Example:
    
    1. Define Services
        
      public interface IPlateService
      {
          Guid PlateId { get; }
      }

      public class PlateService : IPlateService
      {
          public Guid PlateId { get; } = Guid.NewGuid();
      }

    
    2. Register Services in Program.cs
      
      var builder = WebApplication.CreateBuilder(args);

      // One plate for the entire restaurant (singleton)
      builder.Services.AddSingleton<IPlateService, PlateService>();

      // One plate per customer/request (scoped) [NOTE: Each customer is like each request]
      builder.Services.AddScoped<IPlateService, PlateService>();

      // New plate every time you ask (transient)
      builder.Services.AddTransient<IPlateService, PlateService>();

      var app = builder.Build();

      app.MapGet("/plate", (IPlateService singleton, IPlateService scoped, IPlateService transient1, IPlateService transient2) =>
      {
        return new
        {
          Singleton = singleton.PlateId,
          Scoped = scoped.PlateId,
          Transient1 = transient1.PlateId,
          Transient2 = transient2.PlateId
        };
      });

      app.Run();

   3. What Happens When You Hit /plate

   👉 First HTTP request (Customer 1):

   {
    "Singleton": "aaa111",   // same forever
    "Scoped": "bbb222",      // same for this request
    "Transient1": "ccc333",  // new each injection
    "Transient2": "ddd444"   // new each injection
   }


   👉 Second HTTP request (Customer 2):

   {
    "Singleton": "aaa111",   // still same (one plate for restaurant)
    "Scoped": "eee555",      // new plate for new customer
    "Transient1": "fff666",  // new every time
    "Transient2": "ggg777"   // new every time
   }

   🔑 What you see

   Singleton → Same across all requests.
   Scoped → Same within one request, but changes across requests.
   Transient → Always new, even inside the same request.

10. dotnet watch run vs dotnet run?
    ********************************

    🔹 dotnet run

         - Compiles and runs the application once.

         - If you change code, you must stop (Ctrl+C) and run again.

         👉 Example:

               dotnet run
               Starts your app.
               If you edit a controller or class → you must re-run to see changes.

    
    🔹 dotnet watch run

        - Compiles and runs the application, then keeps watching your code files.

        - If you change code (C#, csproj, etc.), it automatically rebuilds and restarts the app.

        - Supports Hot Reload (in .NET 6+), meaning many changes apply instantly without restarting.

        👉 Example:

           dotnet watch run
           Starts your app.
           If you edit a method in a controller → app reloads immediately (no manual stop/start).

    
    🔑 Real-Life Analogy

         - dotnet run → Like playing a movie file. If you edit the file, you have to stop and play again.
         - dotnet watch run → Like watching a movie while the director updates scenes live — you see the changes immediately.


      📝 When to use

         - dotnet run → production-style runs, one-time testing.
         - dotnet watch run → development, when you’re coding and want fast feedback without restarting manually.
         

   - In development, you can just start:

           dotnet watch run
            and leave it running.

     Then:

        - Every time you save a file (.cs, .cshtml, .razor, etc.) →
         dotnet watch detects the change.

        - It tries to apply Hot Reload (for supported changes).

        - If Hot Reload isn’t possible (e.g., you add a new class constructor signature), then it will rebuild and restart the app automatically.


      🔑 What Hot Reload can usually handle (no restart):

          - Change method bodies.
          - Change property getters/setters.
          - Change Razor/Blazor markup.
          - Update local variables and statements.

      
      🔑 What forces a restart:

         - Changing method signatures (parameters/return type).
         - Adding/removing classes or namespaces.
         - Editing Program.cs startup pipeline significantly.
         - Changing project files (.csproj).

      
      💡 Workflow in practice

         - Run dotnet watch run once in the morning.
         - Keep coding → each save updates your running app.
         - You don’t need to restart manually unless you stop coding for the day.

   
   🔹 What happens with dotnet watch run

         - When you run dotnet watch run, your app is running in memory just like dotnet run.
         - On top of that, the watcher process monitors your source files.
         - If a file changes → it triggers Hot Reload or a rebuild + restart.
         - If nothing changes → it just sits idle, consuming very little CPU (mostly sleeping).

   
   🔹 Memory usage

        - The memory being used is basically:
        - Your application running in memory (same as dotnet run).
        - A lightweight watch process (file system watcher).
      💡 So when you’re not coding, the watcher isn’t actively burning resources — it’s waiting. The memory usage is almost the same as if you had started your app with dotnet run.


   🔹 Should you leave it running?

        - During development → yes, it’s convenient to leave dotnet watch run running while you’re coding, since every save updates your app.
        - If you step away for a long time (e.g., lunch break, end of day) → it’s fine to stop it (Ctrl+C) to free up memory and database connections.

    
    🔑 Analogy

        - Think of it like a car engine idling:
        - While you’re actively driving (coding), it’s very handy.
        - If you’re parked for a long time (not coding), you can turn it off to save fuel (memory/resources).

    
    ✅ So:

        - No, it’s not wasteful while you’re coding.
        - If you’re done coding for a while → better to stop it.
